# 详细配置步骤

## 前提条件

- Git 仓库（本地或远程）
- Codex Desktop 已安装并登录

## 你唯一需要做的事

在 Codex 新对话中说：

```
"我要做一个 {项目描述}。目标用户是 {谁在用}，核心功能是 {1-3 个功能}。"
```

## 接下来会发生什么

**你不会一次性拿到最终方案。Architect 和 PM 会反问你**——这不是 bug，这是设计如此。

### 典型的对话过程

```
你：我要做一个面向小团队的工时管理系统

Architect：明白。核心场景是员工填工时、经理审核、生成周报？
          几个问题：
          1. 团队规模大概多大？10人还是1000人？
          2. 需要和飞书/钉钉/企业微信集成吗？
          3. 你更看重开发速度还是长期可维护性？

你：团队 10-50 人，不需要集成，先快速出 MVP

Architect：清楚了。推荐：React + Express + PostgreSQL + Vercel 部署。
          这个方案开发快、免运维、够用到 1000 人规模。
          详细分析如下……你看可以吗？

你：确认

Architect：AGENTS.md 已写入。PM 可以开始写需求了。

PM：第一个功能你想做哪个？工时填报、审核流程、还是报表？
    如果是填报：移动端也要吗？支持补填过去的工时吗？

你：先做 PC 端填报，要能补填

PM：收到。我来写需求……
    作为员工，我想要在 PC 端填报每日工时并支持补填历史日期，
    以便准确记录工作时间。验收标准如下……确认吗？

你：确认

PM → send_message_to_thread → Dev：需求已就绪
```

### 什么情况下会被反问

| 你说的 | 会被反问 |
|--------|---------|
| "做个登录" | 邮箱？手机？第三方？要不要找回密码？ |
| "做个搜索" | 搜什么字段？模糊还是精确？要不要历史记录？ |
| "做个通知" | 站内？推送？邮件？什么事件触发？ |
| "类似抖音" | 你指的是短视频、推荐算法、还是直播？ |
| "要快" | 你是说开发速度快还是页面加载快？ |

## 技术选型逻辑

Architect 不需要你懂技术。你只需描述业务，架构师自己分析：

| 你描述的 | Architect 推断 |
|----------|---------------|
| "内部工具" | 简单技术栈，够用就行 |
| "面向 C 端用户" | 考虑 SEO、CDN、高并发 |
| "处理支付" | 需要事务一致性 → PostgreSQL，不用 MongoDB |
| "实时协作" | WebSocket → Node.js 或 Elixir |
| "数据看板" | 复杂查询 → PostgreSQL，前端图表库 |
| "快速验证想法" | 单体部署，Vercel/Railway，不用 Kubernetes |
